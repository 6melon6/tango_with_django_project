# Diet Tracker 项目介绍

## 项目概述

Diet Tracker 是一个基于 Django 的饮食记录 Web 应用，帮助用户记录每日饮食、追踪卡路里摄入，并提供简单的饮食建议。

---

| 层次 | 技术 | 说明 |
|------|------|------|
| Web 框架 | Django 6.0.2 | 后端框架，MVC 架构 |
| 数据库 | MySQL / SQLite | 生产用 MySQL，开发可用 SQLite |
| 模板引擎 | Django Templates | 服务端渲染 HTML |

| 语言 | Python 3.14 | 后端语言 |
| 时区 | Asia/Shanghai | 中文界面 |

## 功能模块

### 1. 用户认证

**功能**: 注册、登录、登出

**实现方式**:
- 注册使用 Django 内置 `UserCreationForm`，扩展了 email 字段
- 登录使用 Django 内置 `AuthenticationForm`
- 所有受保护页面使用 `@login_required` 装饰器
- 登录成功后自动跳转到仪表盘
- 未登录访问受保护页面自动跳转到登录页

```python
# 注册成功后自动登录
user = form.save()
login(request, user)
return redirect('dashboard')
```

---

### 2. 食物分类管理

**功能**: 创建、编辑、删除食物分类（如主食、肉类、蔬菜）

**实现方式**:
- 模型: `FoodCategory`，字段包括 `category_name`（唯一）和 `description`
- 视图: 标准 CRUD，使用 `FoodCategoryForm`（ModelForm）
- 分类名称有唯一性约束，防止重复

```python
class FoodCategory(models.Model):
    category_name = models.CharField(max_length=100, unique=True)
    description = models.TextField(blank=True)
```

---

### 3. 食物库管理

**功能**: 维护食物信息，支持按名称搜索和按分类筛选

**实现方式**:
- 模型: `Food`，关联 `FoodCategory`（外键），存储每 100 克卡路里
- 搜索使用 Django ORM 的 `icontains` 模糊查询
- 分类筛选通过 `category_id` 过滤
- 建立了两个数据库索引优化查询性能

```python
# 搜索和筛选
queryset = Food.objects.all()
if search:
    queryset = queryset.filter(food_name__icontains=search)
if category:
    queryset = queryset.filter(category_id=category)
```

---

### 4. 饮食记录

**功能**: 记录每日饮食（选择食物、填写数量、选择餐次和日期）

**实现方式**:
- 模型: `FoodRecord`，关联 `User` 和 `Food`，包含数量、餐次、日期、备注
- 卡路里通过模型属性动态计算，不存储在数据库中
- 用户数据隔离：所有查询都加 `user=request.user` 过滤
- 表单验证：数量必须 > 0，日期不能是未来

```python
# 卡路里动态计算
@property
def total_calories(self):
    return (self.food.calories_per_100g * self.quantity) / 100
```

**餐次枚举**:

| 值 | 显示 |
|----|------|
| breakfast | 早餐 |
| lunch | 午餐 |
| dinner | 晚餐 |
| snack | 零食 |

---

### 5. 仪表盘

**功能**: 展示今日饮食汇总、按餐次统计、饼图可视化、饮食建议

**实现方式**:

#### 卡路里计算
使用 Django ORM 的 `aggregate` + `F` 表达式在数据库层面计算，避免 Python 层循环：

```python
total_calories = entries.aggregate(
    total=Sum(F('quantity') * F('food__calories_per_100g') / 100)
)['total'] or 0
```

#### 饮食建议（规则引擎）
纯 Python 规则判断，无 AI/ML：

| 条件 | 建议 |
|------|------|
| 总卡路里 < 1200 | 今天的摄入量偏低，建议均衡饮食 |
| 总卡路里 1200-2000 | 很好！您的摄入量在健康范围内 |
| 总卡路里 > 2000 | 您已超过推荐摄入量，建议选择更清淡的食物 |
| 单餐 > 800 | 您的{餐次}摄入量较高，建议平衡各餐 |

#### 饼图
使用 Chart.js，后端将数据序列化后直接注入模板变量渲染。

---

### 6. 历史查询

**功能**: 按日期或日期范围查询历史记录，支持餐次筛选，折线图展示趋势

**实现方式**:
- 支持三种查询模式：单日、日期范围、默认今天
- 折线图数据通过 `values('record_date').annotate(total=...)` 按日期聚合
- 日期字符串解析使用 `datetime.strptime`，异常时返回空结果

```python
# 日期范围查询 + 折线图数据
chart_data = base_qs.filter(
    record_date__gte=parsed_start,
    record_date__lte=parsed_end
).values('record_date').annotate(
    total=Sum(F('quantity') * F('food__calories_per_100g') / 100)
).order_by('record_date')
```

## 数据模型

```
auth_user (Django 内置)
    │ 1:N
    ▼

    ▼

    │ N:1
```

### 字段定义

**FoodCategory**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BigAutoField | 主键 |
| category_name | CharField(100) | 分类名称，唯一 |
| description | TextField | 描述，可为空 |

**Food**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BigAutoField | 主键 |
| food_name | CharField(100) | 食物名称 |
| category | ForeignKey | 关联分类，级联删除 |
| calories_per_100g | PositiveIntegerField | 每 100 克卡路里 |
| unit | CharField(20) | 单位，默认"克" |

**FoodRecord**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BigAutoField | 主键 |
| user | ForeignKey | 关联用户，级联删除 |
| food | ForeignKey | 关联食物，级联删除 |
| quantity | DecimalField(6,1) | 数量 |
| meal_type | CharField(10) | 餐次枚举 |
| record_date | DateField | 记录日期 |
| notes | TextField | 备注，可为空 |
| total_calories | @property | 动态计算，不存库 |

## 安全机制

| 机制 | 实现方式 |
|------|----------|
| 登录保护 | `@login_required` 装饰器 |
| CSRF 防护 | Django 内置中间件 + `{% csrf_token %}` |
| 用户数据隔离 | 所有查询加 `user=request.user` |
| 密码安全 | Django 内置密码哈希（PBKDF2） |
| SQL 注入防护 | Django ORM 自动参数化查询 |
| 他人数据访问 | `get_object_or_404(FoodRecord, id=id, user=request.user)` |

## 项目亮点

1. **数据库层计算卡路里** — 使用 Django ORM `F()` 表达式在 SQL 层聚合，性能优于 Python 层循环
2. **用户数据完全隔离** — 每个查询都绑定当前用户，无法访问他人数据
3. **卡路里不冗余存储** — `total_calories` 作为 `@property` 动态计算，数据始终一致
4. **规则引擎建议** — 纯 Python 逻辑，无外部依赖，易于扩展
5. **完整测试覆盖** — 80 个测试用例覆盖所有核心路径和边界情况
6. **数据库迁移版本控制** — 通过 Django Migrations 管理表结构变更历史
